Simple 2-condition RNA-Seq differential expression

We are going to do a fairly straight forward RNA-Seq differential expression analysis. We are going to compare two conditions (ESC vs iPS) where we have 3 replicates for each condition. We are going to go through QC, quantitation, differential expression and reviewing of our hits.

The data has already been loaded from the mapped BAM files using default import conditions, and I've cleaned up the sample names a bit, but other than that the data is as you'd see it when you start a project.

Quality Control

Raw data examination

The first thing we want to do is to have a look at some of the raw data. We'd look all around the samples but I'll post some screenshots here so you can see what we're looking for.

Our main concerns are:

1. Is this actually RNA-Seq

2. Are we using the correct assembly

3. Does the data match the annotation

4. Can we see that the mapping is definitely spliced

5. Is there any evidence for DNA-contamination

6. Is there any evidence of PCR duplication

7. Are our libraries directional, and if so are they same-strand or opposing-strand specific.

Let's look at some of these.

From looking at any old random gene we can clearly see that the sample is RNA-Seq. We see clear enrichment of the data around the exons of the gene.

We can see that the library is directional because all of the reads come from the same strand (they're all red, so the top strand in this case). The directionality is opposing strand specific since the feature comes from the bottom strand. If we found a top strand feature we'd see that the reads all came from the bottom strand.

If we zoom in on some of the exons we can clearly see that the imported data is spliced as reads shaply terminate at the end of exons. We can also confirm that we're dealing with the correct assembly since the positions of the reads aligns exactly against the annotation positions.

Looking more widely we can see no evidence for DNA contamination since we see very few reads in intergenic regions or over unexpressed genes. The reads we do see not aligning against features are still directional suggesting that they originate from RNA rather than DNA.

Looking at lowly covered regions we see some instances where we could have PCR duplicates, but these are a minority of all positions, so any duplication appears not to be widespread.

Automated QC

Looking at the RNA-Seq QC plot for this data confirms our other observations. The data all looks nice - most of it is in the exons of genes, it's highly directional (opposing strand specific). We see no rRNA or Mitochondrial sequence and even the amount of data is pretty similar. We could flag and investigate outliers, but that's not an issue here.

Duplication Plots

We can look at PCR duplication in a more systematic way by drawing duplication plots. These plot read density vs duplication. For these plots we need to make probes over all exons, since we need regions with consistent read density.

What we see in these plots is that there is a stong linkage between the amount of duplication and the local read density. This means that although there is duplication here it is biological in nature rather than technical so there is no problem. If we saw high duplication at low read densities then there would have been a problem.

Other QC

We're not going to do any more QC on this data, but other things you should look at would be:

1. Check the sex of the samples (there will be a separate vistory on this topic)

2. Check any expected effects (knockouts or know expected expression changes)

Quantitation

Initial Quantitation

We are going to use the quantitation pipeline to quantitate this data. We're going to do the simplest default quantitation here which is to generate globally normalised, log transformed values, counted against the opposite strand to merged mRNA features. You could do some tweaking of which transcripts you're using to try to clean up your gene set, but I'll leave that for another vistory.

Firstly we need to make up replicate sets for our two groups. Because we have nicely named samples I can do this in a single operation using the Auto Create Sets option.

Pairwise Comparison

Scatterplot

We'll start by simply plotting the two replicate sets against each other to see if they look like there are any changing genes.

There are some things which appear to change, but the changes are relatively modest. We can look at some of the outliers and see what they look like in the raw data.

We can see some of the largest differences which seem to be nice and consistent between replicates, so we should be hopeful of finding some hits when we run the stats. We can also find examples though where the changes are not consistent, and in these cases they seem to often be due to the behaviour of ESC2, which was the odd outlier in the clustering.

Count based statistics

For the first analysis we're going to use the DESeq2 package to find significant differences between our groups. We need to generate raw counts for this test so we'll have to re-run our quantitation first.

The hits seem sensible in that they lie on the outer edge of the distribution of points. We can also see that there are some grey points lying off the main cloud. These should be genes whose expression is not consistent between replicates. We can look at some of these to check.

Again, it's the ESC2 sample which is weird, so this fits with our earlier observation.

We can also look at whether our hits might be being drived by unusual variance in our replicates using the variation plot. We want to see that the variation in a replicate set is the same for the hits as it is for genes as a whole.

In both the ESC (top) and iPS (bottom) groups we can see points which were selected as significant hits, but which had unusual variance. These would not be genes we would want to follow up on since they're probably being driven by variance rather than an actual difference. If we look at some of the biggest outliers then again we can see that ESC2 is the major culprit. The images below are for genes which were selected as significantly changing by DESeq.

Intensity Difference Filtering

DESeq hits are based on the consistency of difference between the two groups, but they don't intrinsically care about the magnitude of the difference. From the scatterplot you can see that the hits extend into the main cloud of points, so the barrier to being significant is just where we ran out of power. We got ~1000 hits here which isn't too bad, but it's too large to follow up individually. We are therefore going to use a second statistical test to identify magnitude of effect outliers, the intensity difference filter.

As the intensity difference filter builds a global model of the relationship between variance and expression we must use the more complete starting list (minus the unexpressed genes) to filter from. If we use the DESeq hit list then we'll get nonsensical results.

Filtered List Intensity Difference p<0.05

Filter on probes in Exoressed anywhere where minimum p-value when comparing ESC to iPS was below 0.05 multiple testing correction applied with a sample size of 187 when constructing the control distributions. Quantitation was RNA-Seq pipeline quantitation on merged transcripts counting reads over exons. Log transformed. Assuming a Opposing strand specific library

With this test you can see that we identify all of the points which separate from the main cloud.

Combining hits

We now have two separate hit lists from the two different statistical filters. Our ideal candidates will be significant in both, so we can combine them to get a final candidate list.

Filtered List DESeq AND Intensity Difference FDR<0.05

DESeq stats FDR<0.05 AND Intensity Difference p<0.05

This scatterplot summarises the hits we got. The ideal hits are in green and they are significant in both consistency and magnitude of effect. The DESeq only hits are in blue - these are still usable and you can take these if you want to expand your hit set for GO analysis for example, but the changes here might be small.

The red points are only significant by intensity difference, meaning that although the changes are big they will not be consistent. Many of these will be due to the odd behaviour of ESC2.

Reviewing hits

Finally, if we were selecting candidates to follow up we'd want to review and prioritise our hits. One problem we have is that in our combined probe list we no longer have a p-value annotated. To fix this we're going to use the List Annotation Value filter to filter (with options which mean that everything will pass) against the FDR from the intensity difference list so we get a new version of our hit list with the FDR value annotated again. We can then make a report and sort by this column.

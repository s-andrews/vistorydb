Promoter Annotation Demonstration

One common action when generating a set of genomic hit probes is to try to relate them to the promoters of genes to gain some functional insight into the hits. There are a few different ways you can do this in SeqMonk and each provides slightly different answers. Here we go through a couple of options to show how you can achieve this.

In this example I'm going to use a project with no real data in it and I'm going to use a set of 10,000 randomly generated 1kb probes as my 'hit list'. I'm then going to annotate the subset of these which are putative promoters for genes.

I've loaded in a fake 1 read dataset and have created the randome probes already so all I'm going to focus on are the annotations.

Direct annotation using existing genes

For the purposes of this demonstration I'm going to limit my definition of promoters to be the region around the start of genes. Obviously you could do the same actions shown here using mRNA features as the basis for the annotation which would give you better coverage where there were alternative promoters, but we're going to simplify things here.

Once we have a set of probes to annotate the quickest way to do this is to use the Annotated Probe Report since this allows you to relate each probe to one feature in a fairly flexible way.

When we construct the report we are asked what the relationship of our feature should be to the probe we're trying to annotate. In our case we want the feature to be just downstream of the probe (so the probe is upstream of the feauture) so I'm going to annotate with downstream genes and allow a 1kb window so that the probe must be very close to the start of the gene. To keep the output more compact I'm only going to show the probes which found an annotation and exclude the rest.

There are however limitations with this approach which might mean that you would prefer to take a slightly more complex method of annotation.

Limitations of the Annotated Probe report

There are two major limitations of the annotated probe report.

1. The report can only annotate a single gene for a given probe. In some cases there will be two (or very occasionally more) genes whose promoters could be associated with the probe.

In this example the probe is annotated to the Gins3 gene, but you can see that it could also potentially be a promoter for the Gm45702 gene, but there is no way to illustrate this in the report.

2. For the annotation to be performed the probe needs to be wholly upstream of the gene. If it overlaps the gene at all then the annotation will not be performed. You can adjust the annotation options when generating the report to say upstream of overlapping, but then you will get annotations which are nowhere near the start of the gene, which probably isn't what you want.

A probe such as this would not be annotated with the previous annotation probe report settings because it overlaps the gene. If we allow 'upstream of overlapping' annotations then we'd also allow hits such as the one below, which wouldn't be ideal.

Annotation via a custom promoter feature track

The alternative way to approach this problem is to create a feature track composed solely of promoter regions and then use this to annotate our probes. This allows for a simpler annotation since the complexity occurs when creating the annotation track, not when annotating the actual probes.

Making the promoter annotation track

To create a promoter annotation track we are going to use the feature probe generator to make probes over promoters and then use this to annotate our future probe set.

The feature probe generator has a lot of flexibility in the relationship that the probes have to the features. This allows us to make fairly tight definitions of what we consider to be promoters.

In general we are going to have two ways to approach this.

1. We can make fairly tight definintions of what consitutes a promoter and then perform our annotation by simply requiring an overlap with these regions.

2. We can make a much broader definition of what consititutes a promoter and then insist that our probe be completely contained within this region.

In this example we're going to do the first of these two options.

For our promoters we're going to say that a promoter is defined as being from 500bp upstream of the transcription start site down to 200bp into the actual gene. We can generate the probes that match this description using the feature probe generator.

Selecting genes

One problem which quickly becomes apparent when you look at the set of probes defined for promoters is that in newer annotation sets for the major genomes there are huge numbers of very small genes which are defined as being biotypes such as TEC (to be experimentally confirmed), lincRNA, pseudogene and similar. You may decide that these are not annotations you wish to pursue, and want to just use more conventional protein coding genes.

If you want to do this you can use the search tool to search for the biotype you want to select and then take the hits and convert them into an annotation track. In this example I'll just select the protein coding genes.

The protein coding genes are a smaller set and can remove some of the noise. This won't be appropriate in all cases, but here it works pretty well. We can remake our promoter probes from this set of genes, and then turn them into an annotation track. You can turn any probe set into an annotation track by right clicking on it and selecting "Convert to Annotation Track"

Dealing with muiltiple hits

The final issue we haven't addressed is what happens if our probe overlaps two promoters. At the moment we get just one of them reported (the first one which was found), which is a bit arbitrary. Since there isn't a nice way to annotate variable numbers of hits to a list of promoters the nicer answer is to perform the annotation the other way around - ie based on the promoters rather than based on the probes.

If we use the Feature Report we can annotate the subset of promoters which overlap with probes, and the same probe can trigger multiple promoters to be annotated. If what we're interested in is just the set of potentially affected genes then this approach will work well.

In the example above the same probe will allow both the Naif1 and Slc25a25 genes to be added to the annotated list since it overlaps with both of their promoters.

Hopefully this summary makes it a bit clearer what your options are when annotating lists of hits and the differences between the different approaches.
